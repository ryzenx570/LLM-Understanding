Original design — Vaswani et al. 2017
The original transformer had two halves: an encoder AND a decoder
The 2017 "Attention Is All You Need" paper designed the transformer for machine translation — turning one language into another. It needed two parts: an encoder to read and understand the source language, and a decoder to write the target language one word at a time. Click any block below to learn what it does.
Encoder-only models
The encoder reads everything — perfect for understanding, not generation
An encoder-only model takes all input tokens simultaneously and produces a rich contextual representation of each one. It sees the full sentence in both directions — every word can attend to every other word with no masking. This makes it exceptional at tasks where you need to understand what text means, but it cannot generate new text word by word.
What it is great at
Text classification, sentiment analysis, named entity recognition, question answering (extractive), semantic search, sentence similarity. Tasks where understanding the full context of all words matters most.
What it cannot do
Generate new text. It has no mechanism to produce one new token at a time. You cannot ask BERT to write a story or answer in a full sentence — it can only label, rank, or extract from existing text.
Decoder-only models — GPT, Llama, Claude, Gemini, Mistral…
The decoder alone — drop the encoder, drop cross-attention, keep everything else
The decoder-only architecture keeps the causal masked self-attention and feed-forward layers, but removes the encoder entirely and therefore removes the cross-attention layer too. The result is a model that excels at one task: given all the text so far, predict the next token. Repeat. This turns out to be general enough to do almost everything.
The causal mask — the key ingredient
A triangular mask hides the future during training and generation
During training, the model sees the entire text but a triangular matrix of −∞ values is added to the attention scores, making future tokens invisible. Token at position 5 gets a score of −∞ for positions 6, 7, 8… — and after softmax, those become 0 (no attention). This one trick means a single forward pass trains on every position simultaneously, making decoder-only models extremely efficient to train while still learning autoregressive generation.
The big insight — one objective rules them all
Predicting the next token turns out to be general enough to solve almost everything
When the original transformer was designed in 2017, most researchers believed you needed a specialized model for each task — one for translation, one for summarization, one for question answering. GPT-1 (2018) challenged this. GPT-3 (2020) proved it wrong entirely. A large enough decoder trained only on next-token prediction can translate, summarize, write code, solve math problems, and hold conversations — all without any task-specific architecture.
Why NOT encoder-only?
Encoders cannot generate
An encoder produces rich representations but has no mechanism to produce new tokens one at a time. You cannot ask BERT to write an essay. The bidirectional attention that makes encoders great at understanding makes them fundamentally unable to generate — they need to see the full sequence before producing any output.
Why NOT encoder-decoder?
Two components, twice the complexity
The full transformer needs two separate stacks and a cross-attention bridge. For translation this is essential — the encoder reads French, the decoder writes English. But for a chatbot responding to a prompt in the same language, it is overkill. The decoder alone, given the prompt as its input sequence, does the same job with half the architecture.
Scaling favors the decoder
One simple objective scales cleanly
Next-token prediction is a single, clean training signal applicable to any text — books, code, conversations, math, news. You never run out of training data. As you scale parameters and data, performance keeps improving. The encoder-decoder design introduced awkward questions about what text the encoder should see vs the decoder — the decoder-only design sidesteps all of that.
Instruction tuning closes the gap
RLHF turns a predictor into an assistant
A raw decoder just completes text — it is not a chatbot. But RLHF (Reinforcement Learning from Human Feedback) fine-tunes it to follow instructions. The result: GPT-4, Claude, Gemini. The underlying architecture is still decoder-only next-token prediction. Only the training signal changes in the final stage.
The three families — a quick summary
Family Architecture Attention type Training objective Best for
Encoder-only BERT, RoBERTa, DeBERTa Bidirectional (no mask) Masked token prediction Classification, search, NER
Decoder-only GPT, Llama, Claude, Gemini Causal (masked future) Next token prediction Generation, chat, reasoning
Encoder-decoder T5, BART, mT5 Bidirectional enc + causal dec Sequence-to-sequence Translation, summarisation
Every major model — and which architecture it uses
The decoder-only design now dominates general-purpose AI. Encoder-only models still thrive in search and classification pipelines. Encoder-decoder models are still the go-to for translation and structured text transformation.

BERT (Google, 2018)
Encoder-only. Masked language modeling. 110M–340M params. Still the backbone of most search engines and classification systems worldwide.
RoBERTa (Meta, 2019)
Encoder-only. BERT trained longer with more data. Stronger on NLP benchmarks. Used in recommendation and ranking systems.
DeBERTa (Microsoft, 2020)
Encoder-only. Adds disentangled attention for position and content. State of the art for many classification tasks.
GPT-2 (OpenAI, 2019)
Decoder-only. 1.5B params. First model to generate convincingly human-like paragraphs. Shocked the world — OpenAI initially refused to release it.
GPT-3 (OpenAI, 2020)
Decoder-only. 175B params. Proved that scale alone produces emergent abilities — code, math, reasoning — with no task-specific training.
ChatGPT / GPT-4 (OpenAI)
Decoder-only + RLHF. GPT-4 estimated ~1 trillion params. The model that brought LLMs to mainstream public awareness.
Claude (Anthropic)
Decoder-only. Constitutional AI training on top of RLHF. Designed with safety and honesty as primary objectives alongside capability.
Llama 1/2/3 (Meta)
Decoder-only. Open weights — anyone can download and run them. Llama 3 rivals GPT-4 in many benchmarks. Uses RoPE positional encoding.
Mistral / Mixtral (Mistral AI)
Decoder-only. Mistral 7B outperforms Llama 2 13B. Mixtral uses Mixture of Experts — only 2 of 8 expert sub-networks activate per token.
Gemini (Google DeepMind)
Decoder-only, multimodal. Trained on text, images, audio, and video simultaneously. Successor to PaLM and LaMDA.
Falcon (TII, UAE)
Decoder-only. Open-source. Uses multi-query attention — faster inference by sharing key-value heads across attention heads.
T5 (Google, 2019)
Encoder-decoder. Frames every NLP task as text-to-text. "Translate English to French: The cat sat" → "Le chat était assis". 11B param version still widely used.
BART (Meta, 2019)
Encoder-decoder. Encoder uses bidirectional attention; decoder is autoregressive. Excellent at summarization and document generation.
mT5 / mBART (Google/Meta)
Encoder-decoder. Multilingual versions supporting 100+ languages. Power most machine translation systems in production today.
Phi-3 (Microsoft, 2024)
Decoder-only. 3.8B params but performs like a 13B model. Trained on high-quality "textbook-like" data rather than raw internet text. Small but mighty.
The trend in one sentence
Every frontier AI lab building a general-purpose assistant — OpenAI, Anthropic, Google DeepMind, Meta, Mistral — has converged on decoder-only transformers as the architecture of choice. The encoder is not gone; it lives on in BERT-style models that power search engines and recommendation systems. But for anything that needs to generate language, the decoder-only approach won.
